# 当前阶段任务：Phase 4 (选区感知、文本替换与安全熔断)
本阶段的核心目标是：赋予 Agent “就地重写”的能力，并通过 Tiptap 的 History 机制保障用户数据安全。
请严格按照以下步骤执行，每完成一步请与我确认：

## Step 1: Tiptap 选区状态锁定 (防失焦丢失)
- **核心痛点**：当用户在编辑器中选中一段文字，然后去右侧点击“发送”或输入框时，Tiptap 很容易失去焦点导致选区（Selection）丢失。
- **解决方案**：在前端维护一个全局或父级状态 `currentSelection`（包含 `from`, `to`, `text`）。监听 Tiptap 的 `onSelectionUpdate` 事件并实时更新它。
- 在发送请求时，前端将 `full_context` 和 `selected_text` 一起打包传给后端。

## Step 2: Rust 接口与 Prompt 升级 (重写意图)
- 修改 Rust 端的 `ask_agent` 命令，接收 `message`, `context`, `selected_text` 三个参数。
- **升级 System Prompt**：增加如下逻辑：“如果用户提供了 `selected_text`（当前选中的文本），你的主要任务是对这段文本进行润色、重写或修改。如果你决定重写，请**只**输出重写后的文本，并严格使用 `<ACTION_REPLACE>替换后的内容</ACTION_REPLACE>` 标签包裹。不要输出原文本，也不要解释。”

## Step 3: 前端拦截器扩展与替换执行 (主刀手术)
- 在右侧 `AgentPanel` 的流拦截器中，增加对 `<ACTION_REPLACE>(.*?)</ACTION_REPLACE>` 的正则全局匹配。
- 提取到内容后，触发 `onActionReplace(content)`。
- 在 `EditorPanel` 中实现替换逻辑：根据 Step 1 保存的 `currentSelection` 的 `from` 和 `to` 位置，调用 `editor.commands.insertContentAt({ from, to }, content)` 进行精准替换。

## Step 4: 撤销机制与视觉反馈 (安全网)
- 确保 Tiptap 注册并启用了 `History` 扩展（`@tiptap/extension-history`）。
- 当 Agent 执行完 `<ACTION_INSERT>` 或 `<ACTION_REPLACE>`，并触发 `agent-stream-end` 时，在 UI 上（比如编辑器右上角或气泡底部）短暂显示一个提示词：“✨ AI 修改完毕，按 Ctrl+Z / Cmd+Z 可撤销”。
- 确保用户按下快捷键时，Tiptap 能够完美撤销刚才 Agent 生成的整段内容。

# 开发纪律（严格遵守）
1. **边界条件处理**：如果 `<ACTION_REPLACE>` 触发时没有有效选区（比如用户取消了划选），默认降级为在光标处 `INSERT`。
2. **多正则冲突**：注意 `INSERT` 和 `REPLACE` 两个正则在同一个流缓冲区中解析时的优先级和状态清理，防止互相干扰。
3. **小步提交**：跑通“选中一段话 -> 发送重写指令 -> 原地变幻成新文本 -> 成功撤销”的完整闭环后，提醒我进行 Git Commit。
