# 当前阶段任务：Phase 16 (认知架构升级：多智能体协作与复杂任务编排)
本阶段的核心目标是：将单体 Agent 升级为“Router + Multi-Agent”架构，实现复杂任务的自动拆解执行，并引入树状的记忆折叠机制以突破百万字上下文限制。
请严格按照以下步骤执行，每完成一步请与我确认：

## Step 1: 语义路由网关 (Semantic Router)
- **痛点**：目前用户必须手动切换 "Copilot/Brain/Doctor" 模式。真正的智能体应该根据用户的输入自动决定调用哪个能力。
- **任务**：在 `agent-harness-core` 中新增 `router.rs` 模块。
- **实现**：
  - 拦截所有的自然语言输入，经过一个极快速的预判处理（可以通过轻量级的本地正则/向量比对，或发送给 LLM 极小 token 的系统提示）。
  - 输出确定的 `Intent` 枚举：`[Chat, Retrieve_Lore, Analyze_Text, Execute_Command]`。
  - 根据 `Intent`，Harness 自动将任务分发给底层的对应 Sub-Agent（如 RAG 引擎或剧本医生）。

## Step 2: 复杂任务编排器 (Planner Agent & DAG)
- **痛点**：对于“帮我根据大纲把这一章写完”这种宏大指令，一次性生成的质量极差，Agent 容易陷入幻觉。
- **任务**：引入 Plan-and-Solve（先计划后解决）机制。
- **实现**：
  1. 新增 `planner.rs`。当识别到复杂任务时，不直接生成正文，而是强制 LLM 输出一个 JSON 格式的任务执行计划 (Task Steps)。
  2. 示例计划：`[{"step": 1, "action": "search_lore", "query": "林墨的武器"}, {"step": 2, "action": "draft_scene", "focus": "打斗"}, {"step": 3, "action": "refine", "style": "阴暗"}]`。
  3. Rust 后端构建一个简单的状态机引擎，遍历执行这个 JSON 数组中的任务，将上一步的输出作为下一步的上下文输入。

## Step 4: 树状记忆折叠 (Hierarchical Context Compression)
- **痛点**：Phase 8 的滑动窗口（2000字）会丢失遥远的细节，但把全书塞进去又会爆 Token。
- **任务**：实现“层级化摘要引擎”。
- **实现**：
  1. 升级 `hermes_memory.db` 或新建数据结构。
  2. **Chunk 级摘要**：当每 2000 字写完时，触发后台异步生成这 2000 字的摘要。
  3. **章节级摘要**：当一章写完时，将本章的所有 Chunk 摘要合并，再生成一个“本章极简摘要”。
  4. **组装**：在请求大模型时，上下文策略变为：`[全书极简摘要] + [紧邻的前三章摘要] + [当前光标前后的 2000 字原文]`。这样即使用户写到第 100 章，Token 消耗也始终稳定在几千以内，且 Agent 绝对不会“失忆”。

## Step 4: 终端 UI 的“思维链”透传 (CoT Observability)
- **任务**：多步推理和执行过程对用户必须透明，否则用户会认为软件卡死了。
- **实现**：在前端消息气泡中，增加对 `Agent State` 的渲染。
  - 当 Planner 在执行时，气泡内渲染出类似 Cursor 或 GitHub Copilot 的多步进度条：
    - [x] 正在分析用户意图...
    - [x] 正在检索设定库 (找到 3 条设定)
    - [ ] 正在撰写草稿 (50%)
    - [ ] 正在自我审查与润色
  - Rust 后端需通过 Tauri Event `agent-chain-of-thought` 实时推送这些状态流转。

# 开发纪律（严格遵守）
1. **JSON Schema 强校验**：Planner 输出的执行计划必须在 Rust 端使用 `serde` 严格反序列化，如果格式错误，必须有 Retry (重试) 机制。
2. **异步队列隔离**：多步编排任务极其耗时，必须使用 `tokio::spawn` 放入后台队列，允许用户点击气泡上的“中止(Cancel)”按钮随时 `abort` 这个异步任务。
3. **小步提交**：跑通语义路由和树状摘要后 commit；跑通复杂的 Plan-and-Solve 状态机后再 commit。
