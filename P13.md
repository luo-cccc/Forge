# 当前阶段任务：Phase 13 (Harness 引擎剥离与通用化封装)
本阶段的核心目标是：解耦。剥离所有硬编码的“小说写作”相关逻辑，将底座重构为一个支持插件化配置的通用 Agent Runtime（运行时环境）。
请严格按照以下步骤执行，每完成一步请与我确认：

## Step 1: Rust Workspace 重构 (引擎与业务分离)
- **任务**：将原先单体结构的 Tauri 后端，拆分为 Rust Workspace 结构。
- **结构设计**：
  - 创建一个新的 lib crate：`agent-harness-core`。将 `vector_db.rs`, `hermes_memory.rs`, LLM 请求流、状态机逻辑全部移动到这里。
  - 原有的 Tauri `src-tauri` 变成一个依赖 `agent-harness-core` 的 binary crate。
- **解耦**：彻底清除 `agent-harness-core` 中诸如 "chapter", "novel", "outline" 这样的业务硬编码词汇，统一抽象为 `Document`, `Project`, `Node` 等通用概念。

## Step 2: 动态上下文注入与 Prompt 模板化 (Dependency Injection)
- **任务**：Harness 引擎不应该知道自己是在“写小说”。
- **重构**：将系统 Prompt、技能萃取的规则定义，从 Rust 源码中抽离出来，变成配置项（Config）。
- **实现**：`agent-harness-core` 初始化时，需要接收一个 `HarnessConfig` 结构体，其中包含：
  - `base_system_prompt` (基础角色设定)
  - `extraction_prompt_template` (技能萃取模板)
  - `max_context_tokens` (滑动窗口大小)

## Step 3: 标准化 Action Registry (工具注册表)
- **任务**：规范化前端与 Agent 之间的“双向动作”通信协议。
- **重构**：在 Rust 核心和前端 TS 中，定义一套严格的 Type/Struct 映射。
  - 将之前手搓的 `<ACTION_INSERT>` 正则提取逻辑，封装成一个标准的前端 SDK 方法，例如 `harnessParser.registerAction("INSERT", callback)`。
  - 确保不仅能提取 XML 标签，还能稳健地提取标准的 JSON Function Call，以便未来灵活切换模型能力。

## Step 4: 打造无 UI 的核心测试沙盒 (Headless Sandbox)
- **任务**：证明这套引擎是真正独立的。
- **实现**：在项目中新建一个极简的前端页面（例如 `Sandbox.html` 或一个极其简单的 React 路由 `/sandbox`），里面没有任何富文本编辑器或小说管理界面，只有一个单纯的输入框和输出控制台。
- **验证**：在这个沙盒里，输入文字，验证 Agent 能否正常调用底层 Memory、进行 RAG 检索、并触发事件流，确保核心引擎完全不依赖外层的 `Tiptap` 或 `Zustand` 状态。

# 开发纪律（严格遵守）
1. **严格类型定义**：前后端交互的事件名（如 `agent-stream-chunk`, `agent-epiphany`）必须抽离为常数枚举，杜绝魔法字符串（Magic Strings）。
2. **保证向后兼容**：重构完成后，现有的“小说 Writer”业务必须依然能够完美运行，只是它的底层调用换成了通过 `agent-harness-core` 暴露的标准接口。
3. **小步提交**：跑通 Workspace 拆分后 commit；跑通沙盒验证后再 commit。这是一次大规模的重构，必须频繁备份。
