# 当前阶段任务：Phase 7 (多章节工程管理与大纲批量生成引擎)
本阶段的核心目标是：将应用从“单文件”升级为“多文件长篇工程”，并引入一个独立的大纲视图（Beat Sheet），让 Agent 能根据大纲节点自动在后台生成对应章节。
此阶段任务繁重，请严格按照以下步骤执行，每完成一步请与我确认：

## Step 1: 本地文件树与多章节路由 (Project Tree)
- **Rust 后端**：利用 `std::fs` 开发一套项目目录管理命令：`read_project_dir(path)`（读取目录下所有 .md 文件并按名称排序）、`create_chapter(title)`、`save_chapter(title, content)`、`load_chapter(title)`。
- **前端 UI**：在界面最左侧（或集成到已有的抽屉栏）增加一个“文件树 (File Explorer)”侧边栏。
- **状态管理**：实现点击侧边栏章节时，保存当前 Tiptap 编辑器内容到本地，并加载新选中的章节内容渲染到编辑器中。

## Step 2: 节拍器/大纲视图 (Outline & Beat Sheet)
- 在编辑器上方或作为独立的 Tab 页，增加一个“大纲视图 (Outline)”。
- 这实际上是一个本地的 JSON 列表，每个节点包含：`chapter_title` (对应物理文件)、`summary` (本章情节概要/节拍)、`status` (空白/已生成/已精修)。
- 提供 UI 允许用户增删改这些大纲节点，并通过 Rust 存入本地如 `outline.json` 中。

## Step 3: 自动化批量生成引擎 (Director Engine)
- 在每个大纲节点卡片上增加一个 `✨ Generate` 按钮。
- **核心逻辑**：当用户点击生成时，前端向 Rust 发起 `invoke("batch_generate_chapter", { outline_node })`。
- **Rust Harness 组装宏大 Context**：
  1. 隐式读取 `lorebook.json`（全局设定）。
  2. 隐式读取上一章的摘要（如果存在），保证连贯性。
  3. 将当前 `outline_node.summary` 作为最强指令。
- **后台生成**：Rust 带着这个宏大 Context 请求大模型。在此期间，不需要在左侧编辑器逐字打字（以免打断用户正在写的其他章节），而是将生成好的完整文本直接通过 `std::fs` 写入对应的本地 `.md` 文件，并将大纲节点状态更新为“已生成”。

## Step 4: 进度感知与异步反馈 (Async UI)
- 由于生成一整章（可能数千字）需要几十秒甚至几分钟，前端必须有良好的异步反馈。
- 当某章节正在后台生成时，文件树和大纲视图中该章节旁边显示一个 Loading Spinner (例如 🔄)。
- 生成完毕后，Tauri Event 触发一个系统通知（Toast），提示“《第X章》已生成完毕”，用户点击文件树即可查看。

# 开发纪律（严格遵守）
1. **防数据丢失**：在切换章节（Step 1）时，务必做好脏检查（Dirty Check）或自动保存（Auto-Save），绝对不能因为切换文件导致用户刚才写的内容丢失。
2. **Rust 异步任务**：Step 3 的后台生成必须在 Rust 的 `tokio::spawn` 异步任务中执行，绝不能阻塞 Tauri 的主线程或前端 UI 的响应。
3. **小步提交**：跑通“文件树切换保存”后 commit 一次；跑通“大纲驱动后台生成”后再 commit 一次。
