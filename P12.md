# Current Stage Mission: Phase 12 (The Architect's View - Entity & Structure Visualization)
Our core text engine is complete. The goal of this phase is to provide the user with visual tools to manage the complexity of a long-form novel. We will build a visual entity relationship graph and an interactive timeline/chapter board.

Please execute the following steps sequentially. Confirm with me after completing each step.

## Step 1: Data Aggregation for Visualization (Rust Backend)
- **Objective:** Create a unified API endpoint to fetch data needed for visual graphs without overloading the frontend.
- **Task:** Create a new Rust command `invoke("get_project_graph_data")`.
- This command should aggregate data from two sources:
    1.  **Lorebook/Entities:** Extract all defined entities (Characters, Locations, Items) and any explicit relationships defined in the Lorebook JSON or `hermes_memory.db`.
    2.  **Chapter Structure:** Read the `outline.json` or the file tree metadata to get the sequence of chapters and their current status.
- Return a structured JSON object: `{ "entities": [...], "relationships": [...], "chapters": [...] }`.

## Step 2: The Entity Relationship Graph (Frontend UI)
- **Objective:** Provide a visual map of how characters and locations interact.
- **Task:** Create a new React component `LoreGraphView.tsx` (perhaps as a new tab alongside the Editor and Outline).
- **Implementation:** Use a lightweight graphing library (like `react-force-graph` or a simple D3.js force-directed graph setup).
- **Behavior:**
    - Nodes represent entities (e.g., "Lin Mo", "The Broken Temple").
    - Edges represent relationships (if extractable from the Lorebook, or simply co-occurrence in the same chapter chunks).
    - Clicking a node should open a side panel showing the entity's full Lorebook entry and a button to "Ask Project Brain about [Entity]".

## Step 3: Interactive Chapter Board / Storyboard (Frontend UI)
- **Objective:** A high-level, Kanban-style view of the novel's structure.
- **Task:** Enhance the existing `OutlinePanel` or create a dedicated `StoryboardView.tsx`.
- **Implementation:**
    - Display chapters as cards in a horizontal or grid layout.
    - Each card shows the Chapter Title, a snippet of the Summary, and its Status (Empty, Drafted, Refined).
    - **Crucial Interaction:** Allow drag-and-drop reordering of these chapter cards.
- **Backend Sync:** When a card is dragged and dropped, immediately send the new order back to Rust to update `outline.json` and rename/reorder the underlying physical `.md` files if necessary to maintain sequence.

## Step 4: The "Agent Analyst" (Connecting Vision to the Brain)
- **Objective:** Use the Agent to analyze the visual structure.
- **Task:** Add a button in the Storyboard view: `✨ Analyze Pacing`.
- **Behavior:**
    - When clicked, send the *entire* sequence of chapter summaries to the LLM via a new Rust command `invoke("analyze_pacing")`.
    - Prompt: "Act as a structural editor. Analyze this sequence of chapter summaries for pacing issues. Identify slow sections, abrupt jumps, or unresolved arcs."
    - Display the result in a modal or a dedicated side panel, perhaps using the "Doctor" UI established in Phase 9.

# Development Discipline (Strictly Observe)
1. **Performance:** The Entity Graph (Step 2) must be optimized. If there are 500 entities, the physics simulation shouldn't freeze the React render thread.
2. **File System Safety:** Step 3's drag-and-drop reordering is dangerous. Ensure you use atomic file renames or temporary files when reordering physical `.md` files to prevent data loss if the app closes mid-drag.
3. **Incremental Commits:** Commit after the Graph View is rendering correctly (Step 2), and again after the drag-and-drop Storyboard is functional (Step 3).
💡 建筑师的见解：
此阶段将应用程序从“智能文本编辑器”转变为真正的故事开发环境（作家的 IDE）。

该react-force-graph库通常是快速在 React 环境中构建美观节点图的最简单方法，但如果性能在性能较弱的机器上成为问题，请确保限制节点数量或使用 2D 渲染而不是 3D 渲染。
