# 当前阶段任务：Phase 14 (商业化基石：安全、监控与崩溃防御)
本阶段的核心目标是：移除开发者视角的配置方式（如 .env），建立面向 C 端用户的安全凭证管理，以及防御级的崩溃和日志收集系统。
请严格按照以下步骤执行，每完成一步请与我确认：

## Step 1: 告别 .env：OS 级密钥安全保险箱 (Keychain)
- **痛点**：大众用户没有 `.env`。API Key 如果明文存本地 JSON 会有被盗风险。
- **Rust 后端**：引入 `keyring` crate。它可以跨平台调用 macOS Keychain、Windows Credential Manager 和 Linux Secret Service。
- **任务**：实现 `set_api_key(provider, key)` 和 `get_api_key(provider)` 命令。让引擎在发起 LLM 请求时，直接从 OS 保险箱读取 Key，而不是环境变量。
- **前端 UI**：开发一个标准的 `Settings` (设置) 模态框，允许用户输入和验证他们的 OpenAI/DeepSeek API Key。

## Step 2: 引入 Tracing 日志系统 (本地黑匣子)
- **痛点**：用户说“软件点生成没反应”，你如果没有日志根本无法排查。
- **Rust 后端**：引入 `tracing`, `tracing-subscriber`, 和 `tracing-appender`。
- **任务**：配置一个按天滚动的本地日志文件记录器（Rolling File Appender），保存在 Tauri 的 `app_log_dir` 下（如 `~/.config/agent-writer/logs/`）。
- 将现有的 `println!` 或 `eprintln!` 彻底替换为 `tracing::info!`, `tracing::warn!`, `tracing::error!`。包括记录 RAG 检索的耗时、LLM 请求的 Token 消耗等。

## Step 3: 全局 Panic 拦截与优雅降级 (Crash Handler)
- **痛点**：Rust 一旦 Panic（比如遇到坏文件或极端越界），exe 会直接闪退，用户体验极差。
- **任务**：在 Rust 的 `main.rs` 最顶层，使用 `std::panic::set_hook` 挂载一个全局异常捕获。
- **实现**：当发生 Panic 时，不要立刻闪退，而是：
  1. 将 Panic 详细信息和 Backtrace 写入 Step 2 的本地日志。
  2. 弹出一个原生的系统警告框（可以使用 `msgbox` 或 `rfd` crate），提示：“应用程序遇到严重错误，已记录到日志中，请重启”。

## Step 4: “一键导出诊断报告”功能 (Bug 追踪器)
- **前端 UI**：在 Settings 面板底部增加一个 `🐞 导出诊断日志 (Export Logs)` 按钮。
- **交互**：点击后，后端打包（Zip）最近 3 天的 `tracing` 日志文件，并触发系统的保存文件对话框，让用户保存到桌面，方便他们作为附件发给你的客服邮箱。

# 开发纪律（严格遵守）
1. **OS 兼容性**：`keyring` 的跨平台编译偶尔在 Linux 上会缺依赖（如 libsecret）。如果你（Claude）在配置编译时遇到 Linux 特有依赖问题，请给出清晰的文档提示。
2. **脱敏处理**：在记录 `tracing::info!` 时，绝对不可以把用户的长篇小说正文或 API Key 明文打印到日志文件中，防止隐私泄露。只记录 Metadata（如长度、Token、状态码）。
3. **小步提交**：跑通 Keyring 凭证管理并从前端成功设置后 commit；跑通日志持久化和导出后再 commit。
