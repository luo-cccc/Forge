# 当前阶段任务：Phase 3 (富文本编辑器与 Agent 动作介入)
本阶段的核心目标是：在左侧接入真正的富文本编辑器（Tiptap），并在前后端建立一套“上下文注入”与“Agent 逆向操作编辑器”的机制。
请严格按照以下步骤执行，每完成一步请与我确认：

## Step 1: 引入 Tiptap 编辑器内核
- 安装依赖：`npm install @tiptap/react @tiptap/pm @tiptap/starter-kit`
- 在左侧区域新建 `EditorPanel.tsx` 组件，替换掉原来的 `textarea`。
- 配置基础的 Tiptap 实例，包含加粗、斜体等最基本的 StarterKit 功能，并确保其占满左侧屏幕且样式美观（利用 Tailwind 的 `prose` 类）。

## Step 2: 状态共享与上下文提取 (眼睛)
- 建立左右双栏通信的机制。为了简单高效，可以引入一个轻量级的全局状态（如 Zustand）或在父组件（如 `App.tsx`）中使用 `useRef`。
- 目标：当用户在右侧 `AgentPanel` 点击发送消息时，前端能够获取当前左侧 Tiptap 编辑器里的**全部纯文本内容**（作为全局上下文），或者**光标所在段落的文本**（作为局部上下文）。
- 修改 Rust 端的 `ask_agent` 命令签名，除了接收用户的 `message`，还要接收一个 `context` 字符串。

## Step 3: Prompt 工程与工具动作约定 (大脑)
- 在 Rust 的 Harness 状态机中，修改发送给 LLM 的 System Prompt。
- 告诉 LLM：“你是一个写作助手，当前用户的草稿内容是：[{context}]。如果你决定直接为用户撰写或修改内容，请严格按照特定格式输出，例如使用 `<ACTION_INSERT>要插入的文本</ACTION_INSERT>` 包裹你想写入编辑器的内容。”
- *注意：为了降低初期复杂度，Phase 3 我们先不使用复杂的 OpenAI Function Calling 格式，而是采用更易于在流式传输中正则提取的 XML 标签动作法。*

## Step 4: 动作拦截与编辑器逆向控制 (手)
- 在右侧的 `AgentPanel` 接收流的逻辑中，增加一个拦截器：
  - 如果检测到返回的流字符串中包含了 `<ACTION_INSERT>` 的内容，在 UI 上不要把它当成普通聊天展示出来，而是将其提取出来。
  - 触发一个特定的前端动作（例如调用 Tiptap 的 `editor.commands.insertContent(提取的文本)`），让这段文字自动在左侧编辑器的光标位置“敲”出来。

# 开发纪律（严格遵守）
1. **模块解耦**：`EditorPanel` 和 `AgentPanel` 的逻辑必须独立，中间通过清晰的状态管理或事件机制通信，不要写成意大利面条代码。
2. **渐进验证**：Step 1 的编辑器渲染出来后，必须先在本地输入测试一下样式，再进行后续的上下文打通。
3. **小步提交**：跑通“Agent 自动在左侧编辑器输入文字”的魔法时刻后，提醒我进行 Git Commit。
