# 当前阶段任务：Phase 5 (全局设定库与 ReAct 自主检索循环)
本阶段的核心目标是：建立本地的小型知识库（Lorebook），并让 Agent 学会通过 `<ACTION_SEARCH>` 标签主动检索设定，实现多步思考流（Thinking Loop）。
请严格按照以下步骤执行，每完成一步请与我确认：

## Step 1: 本地设定库持久化 (Rust fs)
- 在 Rust 后端，使用 `tauri::api::path::app_data_dir` (v1) 或 `app_data_dir` (v2) 以及 `std::fs`，实现一个简单的本地 JSON 存储逻辑。
- 创建文件 `lorebook.json`，结构形如：`[{ "id": "1", "keyword": "林墨", "content": "主角的师弟，擅长用毒，性格极其腹黑。" }]`。
- 暴露两个 Tauri Command：`get_lorebook()` 和 `save_lore_entry(keyword, content)`。

## Step 2: 前端设定抽屉栏 (UI)
- 在 UI 边缘（例如界面最右侧或左侧）添加一个可折叠的 Drawer 或侧边栏，名为 "Lorebook (设定集)"。
- 提供简单的表单，允许用户添加和删除词条（Keyword 和 Content），并调用 Step 1 的命令持久化到本地。

## Step 3: Prompt 升级与搜索动作约定
- 更新 Rust 端的 System Prompt：“你是一个带检索能力的小说 Agent。如果你需要了解某个设定的详情，请输出 `<ACTION_SEARCH>关键词</ACTION_SEARCH>`，系统会将查到的设定返回给你。在没有拿到设定前，请不要强行编造剧情。”

## Step 4: ReAct 拦截与回传循环 (The Loop - 核心难点)
- 修改后端的 `ask_agent` 或前端的拦截逻辑，建立一个“拦截 -> 检索 -> 重新请求”的机制。
- **推荐在后端 Rust 实现此循环**：
  1. Rust 接收 LLM 的 SSE 流。
  2. 如果流中完整解析出 `<ACTION_SEARCH>林墨</ACTION_SEARCH>`，**立即中断当前流**。
  3. Rust 在本地 `lorebook.json` 中检索 "林墨" 的内容。
  4. Rust 构造一条新的 System 消息（例如：“系统返回搜索结果：[林墨的设定内容]”），将其追加到对话上下文中。
  5. Rust **自动发起第二次 LLM API 请求**，并将新一轮的流继续推给前端。
- 前端对这一切是无感的，前端只需要在气泡里显示状态“🔍 正在检索设定: 林墨...”，然后继续接收并渲染最终的文本。

# 开发纪律（严格遵守）
1. **单向依赖**：前端只管渲染流和动作，多轮请求的闭环尽可能放在 Rust 端，以保证性能和避免网络波动导致的多次握手。
2. **状态透传**：当 Rust 截获 Search 并发起二次请求时，需要通过 Tauri Event 给前端发送一个特殊的提示状态，让气泡 UI 有感知，避免用户以为应用卡死了。
3. **小步提交**：跑通“查字典 -> 注入上下文 -> 继续写作”的完整链路后，提醒我进行 Git Commit。
