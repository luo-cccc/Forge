# 当前阶段任务：Phase 8 (架构夯实与健壮性打磨)
本阶段不增加花哨的新功能，核心目标是偿还技术债：优化前端状态管理、处理大模型 Token 上限、增加原子级数据安全策略。
请严格按照以下步骤执行，每完成一步请与我确认：

## Step 1: 引入 Zustand 进行全局状态重构 (前端重构)
- **痛点**：目前 `EditorPanel`, `AgentPanel`, `OutlinePanel` 之间可能存在大量的 prop drilling（属性透传）或者凌乱的事件监听。
- **任务**：安装 `zustand`。创建一个 `useAppStore`，将以下核心状态抽离到全局：
  1. `currentChapter`: 当前正在编辑的文件名/路径。
  2. `isAgentThinking`: 控制全局的 AI 思考状态（防止重复提交）。
  3. `outlineData`: 大纲节点的缓存数据。
- 重构现有组件，使用 Zustand store 替代深层的 useState，并集中管理 Tauri 的 `listen` 事件的订阅与卸载 (unlisten)，防止内存泄漏。

## Step 2: 上下文滑动窗口与 Token 截断 (Rust 后端优化)
- **痛点**：在 Phase 7 的后台生成中，如果之前写了 50 章，全部塞给 LLM 会导致 Token 超限报错或极高的 API 费用。
- **任务**：在 Rust 端的 `lib.rs` 中引入简单的 Token 限制策略。
  - **滑动窗口**：当发送 `full_context` 给 LLM 时，如果全文过长，默认只截取光标前后（或文章末尾）各 2000 个字符。
  - **大纲精简**：后台生成引擎组装 Context 时，不再加载所有历史大纲，只加载“全局 Lorebook” + “紧邻的前两章 Summary”。

## Step 3: 原子写入与自动保存安全网 (数据安全)
- **痛点**：如果 Rust 使用 `std::fs::write` 写大文件的过程中应用崩溃或断电，会导致 `.md` 文件直接损坏变为空白。
- **任务**：重写所有的文件保存逻辑（包括 Editor 的手动保存和 Director 的后台写入）。
  - **原子写入 (Atomic Write)**：先将内容写入到一个临时文件 `{chapter_name}.tmp`。写入成功并落盘后，再使用 `std::fs::rename` 将其重命名为真正的 `.md` 文件覆盖旧版。
  - **前端防丢**：在 `EditorPanel` 中增加 `debounce`（防抖）自动保存逻辑（例如停止打字 3 秒后自动触发 Rust 的保存命令），并在 UI 右下角显示 "Saved at 10:42" 的微小提示。

## Step 4: 优雅的错误捕获与重试机制 (网络容错)
- **痛点**：大模型 API 经常因为网络抖动超时，目前可能会导致前端状态一直卡在 "Generating" 或是 "⚙️ Thinking..." 转圈。
- **任务**：
  - 在 Rust 端给 `reqwest` 的 LLM 请求增加 `timeout`（例如 60 秒），并捕获 API 返回的非 200 状态码（如 429 Rate Limit）。
  - 如果请求失败，通过 Tauri 发送特殊的 `agent-error` 事件。
  - 前端捕获后，重置状态，并在 UI 上弹出红色的 Toast（例如："API Timeout. Please check your network or try again."），允许用户一键重试。

# 开发纪律（严格遵守）
1. **不破坏现有逻辑**：在用 Zustand 重构时，一定要保证先前的 `Cmd+K`、Tiptap 选区撤销等核心逻辑完好无损。
2. **Rust 错误处理**：全面消灭 Rust 代码中裸露的 `.unwrap()`，改用 `Result` 返回或规范的错误处理，防止后端进程 Panic 崩溃。
3. **小步提交**：每完成一个 Step（特别是 Step 1 的状态重构和 Step 3 的数据安全），强烈建议提醒我进行 Git Commit。
