# 当前阶段任务：Phase 10 (项目大脑与 RAG 全局语义检索)
本阶段的核心目标是：为整个项目引入本地的 RAG (检索增强生成) 机制，建立“全局向量记忆”，让用户可以向 Agent 提问关于整本书的任何细节。
为了保持 exe 的轻量级，我们将手写一个纯 Rust 的内存级向量比对库，不引入沉重的第三方向量数据库。
请严格按照以下步骤执行，每完成一步请与我确认：

## Step 1: 轻量级向量存储引擎 (Rust 纯手写)
- 在 Rust 后端，新增一个模块 `vector_db.rs`。
- 定义结构体 `Chunk { id: String, chapter: String, text: String, embedding: Vec<f32> }`。
- 实现一个纯函数 `cosine_similarity(a: &[f32], b: &[f32]) -> f32` 计算余弦相似度。
- 实现向量库的序列化与落盘，将其保存为项目目录下的 `project_brain.bin` 或 `project_brain.json`。

## Step 2: 自动分块与向量化 (Auto-Embedding)
- 修改我们在 Phase 8 实现的 `save_chapter` 逻辑。
- 当章节保存触发时，增加一个后台异步任务：
  1. 将当前章节的纯文本按段落（如以 `\n\n` 分隔，或每 500 字滑动截断）进行 Chunking。
  2. 调用大模型供应商的 Embedding API（如 OpenAI 的 `/v1/embeddings`，模型如 `text-embedding-3-small`）。
  3. 将获取到的 `Vec<f32>` 和对应的 `text` 更新到 `vector_db` 中。

## Step 3: 全局问答 UI (Chat with Book)
- 在右侧边栏（AgentPanel）增加一个模式切换开关 (Toggle)：`[Copilot 伴写模式] / [Project Brain 全局问答]`。
- 在 Project Brain 模式下，右侧气泡样式可以微调（例如变成紫色主题），强调当前是在和“整本书”对话，而不是在写小说正文。

## Step 4: RAG 检索与生成闭环 (The RAG Pipeline)
- 新增 Rust 命令 `invoke("ask_project_brain", { query })`。
- **检索阶段**：
  1. 将用户的 `query` 调用 Embedding API 转化为向量。
  2. 遍历 `vector_db` 计算余弦相似度，提取 Top 5 最相关的 `Chunk`。
- **生成阶段**：
  3. 组装 System Prompt：“你是一个极其熟悉本小说的专家。基于以下从全书中检索出的原文片段，回答用户的问题：[Top 5 Chunks 组合文本]”。
  4. 发起正常的 LLM 流式请求，通过 SSE 推送给前端的聊天气泡。

# 开发纪律（严格遵守）
1. **API 独立性**：Embedding API 和 Chat API 可能是不同的端点，确保在 `.env` 中正确配置相关的 KEY 和 URL。
2. **轻量与性能**：一本 100 万字的小说，按 500 字切块，也就 2000 个向量。在 Rust 中遍历 2000 个长度为 1536 的数组计算点积，耗时不到 1 毫秒，**千万不要引入巨大的第三方数据库（如 Qdrant/Milvus 客户端），保持打包体积小巧**。
3. **小步提交**：跑通“余弦相似度计算与本地向量落盘”后 commit；跑通“前端提问并召回正确段落”后再 commit。
💡架构师叮嘱：
这绝对是整个架构中最极客感的一步。我们不用花里胡哨的庞大数据库，直接用 Rust 复制的集群算力（SIMD 甚至都不需要，数据量小说对来说足够小），就能实现千年级的全书检索。这种专注于场景的美学，就是桌面级客户端的魅力。
