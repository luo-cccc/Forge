# 当前阶段任务：Phase 9 (AI 剧本医生与行内批注系统)
本阶段的核心目标是：赋予 Agent 审阅长文的能力，不仅要能发现问题，还要能在左侧富文本编辑器中通过“高亮+侧边批注”的形式（类似 Google Docs）将问题精准映射到原文上。
请严格按照以下步骤执行，每完成一步请与我确认：

## Step 1: 引入 Tiptap Annotation/Comment 扩展 (前端基建)
- 开发或配置一个 Tiptap 的批注扩展（可以基于 Mark 实现，例如 `CommentMark`）。
- 该 Mark 需要存储一个 `commentId` 属性。
- 在编辑器 UI 层面，带有 `CommentMark` 的文本应显示为黄色底色（例如 `bg-yellow-200`）。当用户鼠标悬浮或点击该文本时，触发事件。

## Step 2: 剧本医生面板与 JSON 结构化输出 (后端通信)
- 在右侧侧边栏增加一个 "Script Doctor (剧本医生)" 的 Tab 页。
- 新增一个 Rust 命令 `invoke("analyze_chapter")`。
- **核心 Prompt 升级**：在这个模式下，要求大模型必须输出严谨的 JSON 数组（可以使用 OpenAI 的 `response_format: { type: "json_object" }` 或强力 System Prompt 约束）。
- JSON 结构约定：
  `{ "reviews": [ { "quote": "原文中的某句话", "type": "logic|ooc|pacing", "issue": "发现的问题描述", "suggestion": "修改建议" } ] }`

## Step 3: 原文精准映射算法 (核心难点)
- 当前端收到 Rust 返回的 JSON 审查报告后，需要遍历 `reviews` 数组。
- **映射逻辑**：编写一个纯函数，在当前 Tiptap 的纯文本文档中查找 `quote`（原文片段）的起始和结束索引 (`from`, `to`)。由于 LLM 返回的 quote 可能有轻微偏差或省略，如果精确匹配失败，可以使用简单的字符串相似度查找，或者回退到段落级匹配。
- 匹配成功后，针对该 `from` 到 `to` 的区间，自动调用 Tiptap API 施加 Step 1 的 `CommentMark`，并分配一个唯一的 UUID。

## Step 4: 批注交互与一键采纳 (UI 闭环)
- 将生成的批注卡片展示在右侧的 "Script Doctor" 面板中（或者以悬浮卡片形式显示在编辑器右侧空白处）。
- **联动**：点击右侧批注卡片，左侧编辑器自动滚动到对应的黄色高亮处。
- **一键修复**：在批注卡片上增加一个 `✨ Apply Fix` 按钮。点击后，将 `suggestion` 内容作为上下文，调用我们之前写好的 Phase 4 `<ACTION_REPLACE>` 机制，让 Agent 自动把那段黄色高亮的文字替换并重写。

# 开发纪律（严格遵守）
1. **JSON 解析防崩溃**：LLM 输出的 JSON 有时不完美。在 Rust 端必须使用 `serde_json` 进行安全的序列化/反序列化，如果解析失败，向前端抛出优雅的错误而不是直接 Panic。
2. **纯函数查找**：Step 3 的字符串映射算法要考虑到文档可能包含空行、特殊符号。
3. **小步提交**：跑通“生成审查 JSON 并在左侧成功打上黄色高亮”后，立刻提醒我进行 Git Commit。
💡架构师叮嘱：
这个阶段的核心难点在于步骤3的“精确映射”。大模型有时候会自聪明作，把你的译文缩写（比如输出quote“林墨走进房间...叹了口气”）。这导致了用精确的indexOf不够这句原话，从而打不上高亮。你可以提醒Claude Code在写匹配算法的时候，尽量匹配quote的前10个字和后10个字，或者直接让LLM在输出JSON当时顺便返回了它引用的原文是第几段，这样定位会生日。
